Cicada Working Agreement

Team Meeting Etiquette & Frequency

  1. We agree to be available Tuesday and Thursday between 11am and 1pm for working together in person on sprints.
  2. We are to be present and active at meetings.
  3. If something comes up and we can no longer attend a team meeting (or will be late past 5 minutes), give the team know ahead of time.
  4. We agree to check discord a minimum of once a day Monday through Friday.

Team Guidelines

  1. We agree to work hard and aim for a high grade.
  2. We agree to work ahead and avoid procrastination to the best of our ability.
  3. We agree that functionality is more important than making things gorgeous.
  4. We agree to work with safe and dependable methods while allowing ourselves some minor exploration to try new techniques
  5. We encourage each other to admit mistakes, since we're all here to learn.
  6. There is to be no berating, put downs, or blaming/pointing fingers. Instead of dwelling on the mistake, we'll work together to fix and solve mistakes.
  7. We'll note what went wrong and what we did to fix the issue so that we can avoid making the same mistake again.
  8. Resolving Team Conflict
    1. When a conflict occurs during our team, we will message via Discord for minor conflicts and in person for bigger conflicts.
    2. If things escalate beyond a point where we're able to resolve it or is not resolved within a message, we will hop on a virtual call. Beyond that call, if it persists, request Tyler for mediation/help/advice.

Team Processes

  1. Github
    1. Open up a PR, no matter how small the change appears.
      1. In the PR title, note what the PR is about.
      2. In the Description, add in screenshots of what you see (or even a small mov/gif about the behavior).
    2. Have at least one approval from someone else before merging.
    3. You can merge in another person’s PR as long as there’s another pair of eyes - make sure to give a shoutout in Discord.
  2. cPanel
    1. We do not touch any code in the repository or public_html folders in cPanel (it doesn’t matter if it’s adding or deleting or even one small character change, since this will break our automatic deployment script).
  3. Front End Styling
    1. When working on a feature that requires front end styling, discuss with the team to decide on a standard.
  4. Coding Standards
    1. Each tab should be 4 spaces.
    2. When copying and pasting code, make sure everything works as expected and clean up old code/comments that are not appropriate.
    3. When writing CSS, making sure to target a class or id (rather than a global element) to prevent changing someone else's styles. If needed, search through the project for what you're using to make sure you're not impacting another elements style.
